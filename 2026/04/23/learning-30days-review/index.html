30 天学习计划复盘：慢一点，反而更快

发表于2026-04-23|更新于2026-04-23|学习
|浏览量:

这 30 天里，我没有追求“每天学很多”，而是把重点放在“每天都学”。结果比过去冲刺式学习更稳定。

我做对的三件事

  • 每天固定 90 分钟深度学习。
  • 每周做一次总结，不积压问题。
  • 用输出倒逼输入，每学完一块就写小结。

我踩过的坑

  • 目标定太大，容易中途泄气。
  • 只收藏不实践，看起来很努力，实际上没吸收。
  • 不做复盘，重复犯错。

下一步

接下来我会把学习目标拆得更细：按周设目标、按天看执行，用“完成率 + 质量”双指标评估，而不是只看时长。

上一篇
我的学习笔记系统：从“囤积”到“可复用”
过去我记了很多笔记，但真正能用上的很少。后来我把笔记结构改成“问题 -> 结论 -> 示例 -> 场景”，利用率明显提高。 一条笔记最少包含什么 这个问题是什么。 为什么会这样。 一个最小可运行示例。 未来在哪些场景可以复用。 如何避免“笔记坟场” 每周整理一次，把碎片合并。 给每条笔记加标签，方便检索。 学完后 24 小时内至少复现一次。 总结好笔记不是“记得多”，而是“下次能直接拿来用”。当笔记开始服务实战，学习效率才真正提升。
下一篇
写给个人项目的 API 设计小清单
个人项目做久了，我越来越相信：API 先规范，后开发，返工会少一半。 先定三件事 统一响应结构（code、message、data）。 统一错误码，避免前端猜测错误原因。 统一分页参数，减少页面适配成本。 命名建议 资源名用复数，如 /posts、/users。 操作用 HTTP 动词表达，不要把动词塞进路径。 版本号放在前缀，比如 /api/v1。 结论API 的一致性比“聪明设计”更重要。简单、稳定、可预期，才是长期维护最友好的方案。
